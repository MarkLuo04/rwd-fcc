Introduction

Git is a distributed version control system that helps developers track changes in their codebase over time. Unlike centralized systems, Git allows each user to have a full copy of the repository, enabling offline work and better redundancy.

GitHub is a web-based platform built on top of Git. It provides hosting for repositories, along with collaborative features like issue tracking, pull requests, and project boards. Developers use GitHub to share their code, contribute to open-source projects, and collaborate with team members.

Version control helps developers manage project history, experiment safely through branches, and work with others without overwriting each other’s work. It’s a core tool for modern software development, essential for both solo projects and large-scale team collaborations.

Git Configuration and Setup

Before you start using Git, it’s important to configure your identity and set up basic settings. This ensures that your commits are properly attributed and that you can connect to remote repositories securely.

  • git config --global user.name "Your Name": Sets your name in all commits.
  • git config --global user.email "your.email@example.com": Sets your email in all commits.
  • ssh-keygen -t ed25519 -C "your.email@example.com": Generates an SSH key for secure GitHub authentication.

You can also configure other settings like line endings or set up handy aliases to simplify common commands.

Basic Git Commands

Before working with Git, you need to understand some foundational commands:

  • git init: Initializes a new Git repository in your current directory, creating a .git folder that tracks all changes.
  • git status: Shows the current state of your working directory, including changes staged for commit or untracked files.
  • git add: Stages changes, telling Git which modifications you want to include in your next commit.
  • git commit: Saves a snapshot of the staged changes, recording them permanently in the project history with a commit message.
  • git log: Shows a list of past commits, helping you trace the project’s evolution and review past changes.

These commands are the backbone of Git workflows and form the foundation for more advanced operations.

Using .gitignore

The .gitignore file tells Git which files or directories to ignore. This is useful for excluding temporary files, logs, or system-generated content that shouldn’t be tracked in the repository.

  • Common entries include node_modules/ for Node.js projects or *.log for log files.
  • Edit the .gitignore file in the root of your repository and add paths or patterns.
  • If you accidentally tracked a file before adding it to .gitignore, remove it with git rm --cached <file>.

Keeping your repository clean and focused helps avoid bloating it with unnecessary files.

Branching and Merging

Branches allow you to work on different features or fixes without disturbing the main codebase.

  • git branch: Lists all branches or creates a new one.
  • git checkout: Switches between branches so you can work on multiple tasks in parallel.
  • git merge: Brings changes from one branch into another (usually into main or master).

Branching is powerful because it enables safe experimentation — if something breaks, the main line of development remains stable. Just remember to regularly merge and resolve conflicts to keep branches from diverging too far.

Working with Remotes

A remote repository (like on GitHub) lets you share your code with others.

  • git remote add: Links your local repository to a GitHub repository.
  • git push: Uploads your commits to the remote.
  • git pull: Fetches the latest changes and merges them into your local branch.
  • git fetch: Retrieves updates without merging, letting you review them first.
  • git clone: Makes a full copy of a remote repository onto your local machine.

Using remotes ensures that your work stays backed up and synchronized with the team’s collective progress.

Resolving Merge Conflicts

Merge conflicts occur when Git cannot automatically reconcile changes between branches. This usually happens when two branches modify the same lines in a file.

To resolve a merge conflict:

  • Open the conflicting files and edit them to combine the changes.
  • After fixing, use git add <file> to mark the conflict as resolved.
  • Finally, use git commit to complete the merge.

To avoid frequent conflicts, communicate with your team, pull regularly, and keep branches short-lived.

Pull Requests and Collaboration

Pull requests (PRs) are a collaboration feature on GitHub that let you propose changes to a repository.

After pushing your branch to GitHub, you can open a pull request to:

  • Show others the changes you’ve made
  • Request feedback or reviews
  • Start a discussion about the modifications

Other contributors can comment, suggest improvements, or approve the PR. Once approved, the changes can be merged into the main branch.

Best practices include writing clear, descriptive PR titles, linking relevant issues, and being responsive to requested changes. Pull requests help maintain code quality and ensure that all changes are reviewed before being integrated.

Git Revert and Reset

Sometimes you need to undo changes. Git offers two main ways: git revert and git reset.

  • git revert <commit>: Creates a new commit that undoes the changes from a previous commit. This is a safe option because it preserves project history.
  • git reset --hard <commit>: Resets your repository to a specific commit, discarding all commits and changes after it. Use this carefully, as it can permanently remove work.

Understanding when to use revert vs. reset helps you recover gracefully from mistakes.

Git Tags and Releases

Tags are used to mark specific points in your commit history, often to label releases or important milestones.

  • git tag v1.0.0: Creates a new tag called v1.0.0 on your current commit.
  • git push origin v1.0.0: Pushes the tag to the remote repository so others can see it.
  • On GitHub, you can also create releases linked to tags, adding release notes and downloadable assets.

Tags and releases help teams coordinate and communicate about stable versions of their projects.
